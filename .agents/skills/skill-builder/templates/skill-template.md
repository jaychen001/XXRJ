# Skill 骨架模板

创建新 Skill 时，复制下面的骨架，把方括号占位符换成实际内容，删掉用不到的可选段。
样例本身就是合规的 SKILL.md 写法：`[标题]` 段、四空格缩进、纯文本、不加粗、不用符号清单、frontmatter 只 name + description。照它的体例写，新 Skill 自然和现有技能一个手笔。

---

## 必须有的段

```markdown
---
name: [skill-name]
description: 一句话说清什么时候触发、做什么、产出什么
---

[任务]
    一句话说清这个 Skill 做什么。有多种模式就分别说。

[依赖检测]
    启动第一步。列必需和可选，说清缺了怎么办。
    必需：[前置文件或工具] → 缺失则提示引导或自主安装。
    可选：[可选依赖] → 缺失则标降级模式继续。

[文件结构]
    [skill-name]/
    ├── SKILL.md            # 本文件
    ├── references/         # 可选，内容厚时下沉，按需加载
    │   └── [参考].md       # 一句话说明
    └── templates/          # 可选，有产出文档才有
        └── [模板].md       # 输出模板
    单文件 skill 只列 SKILL.md，注释里标无 references / templates。

[第一性原则]
    [原则名]：一句话说明。
    [原则名]：一句话说明。
    联网优先：涉及外部库、API、版本先 WebSearch 再动手。

[工作流程]
    [第一步]
        具体做什么。
    [第二步]
        具体做什么。

[初始化]
    执行 [依赖检测]。
```

## 推荐有的段

```markdown
[输出风格]
    一句话定调：这个 Skill 用什么语气说话，肯定什么、禁止什么。
    典型表达：
    - "示例句一"
    - "示例句二"

[维度清单]
    按领域命名，比如需求维度清单、审查维度清单、开发规则清单。
    列出这个 Skill 要盯的所有维度或规则。

[策略]
    按领域命名，比如对话策略、审查策略、开发策略。
    描述方法论，怎么做。
```

## 按需有的段

```markdown
[充足度判断]
    采集或分析型用。判断信息何时够了可以输出。
    必须满足：[条件]。
    尽量满足：[条件]。

[回退策略]
    发布或部署型用。出问题怎么回退。

[Phase 完成度判断]
    开发型用。一个 Phase 算完成的验证标准。

[references]
    内容厚时用。把问题库、原则、长流程下沉到 references/ 按需加载，SKILL 正文只留指针。
    - references/[文件]  一句话说明
```

## 规范速查

| 项 | 规范 |
|----|------|
| Skill 名 | kebab-case，例 skill-builder |
| 目录位置 | .agents/skills/[skill-name]/ |
| 主文件 | SKILL.md |
| 模板文件 | templates/ 子目录 |
| 段标题 | [标题] 格式 |
| 内容缩进 | 四空格 |
| frontmatter | 只有 name 和 description |
| 语言 | 中文 |
| 示例 | 只放能当标杆的，潦草的宁可不放 |
| 括号 | 不用括号塞补充逻辑，写成正文 |

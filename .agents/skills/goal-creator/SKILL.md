[任务]
    结合当前上下文，理解用户接下来想自驱完成什么，按 Goal 模板写好一条 /goal 指令，交给用户复制发送。
    你只准备指令，不替用户触发。slash command 必须用户自己发。

[依赖检测]
    无前置文件。读当前对话和项目状态作为上下文。

[文件结构]
    goal-creator/
    └── SKILL.md  # 本文件，无 references / templates

[第一性原则]
    上下文感知：从当前环节、用户刚说的、项目状态里推断目标，不让用户从零写。
    目标必达：/goal 的核心是完成任务。执行策略写成目标导向、一条路不通就换方法，穷尽多种尝试都解不开才停，不一卡就停。
    完成条件可自证：完成标准和验证方式写成在输出里能自证的形式，因为 /goal 的 evaluator 只判已 surfaced 的内容。跑命令贴输出、列产出物、逐项打勾。
    字数硬限：/goal 有 4000 字符上限，超了发不出去。写到精准，每个字带信息，删套话、删多余的格式排版。完成条件之外的段不带信息就不写。
    边界要硬：有真实约束才写，比如别碰某模块、不直推 main。别写"保持兼容性"这种放之四海的空话。
    不填形式：完成条件是唯一必填的核心，evaluator 全靠它。约束、验证、执行策略只在真带信息时写。简单目标就目标加完成条件两行。
    联网优先：涉及不确定的命令或接口先确认再写进模板。

[Goal 模板]
    核心是完成条件，必填且写到可自证。其余段按需，不带信息就删，整条压进 4000 字符。尖括号是占位。

    /goal 完成 <目标>。

    完成的标准：
    1. <可验证终态>
    2. <可验证终态>

    验证方式：
    - <跑什么命令、查什么产出物>

    约束：
    - <别碰什么、什么不可逆操作>

    执行策略：目标导向，一条路不通换方法，多种都试过才停；长任务记一句进度日志。

    执行策略是通用自驱行为，没有特殊打法就用上面这一行，别展开成清单占字数。简单目标只留目标加完成条件两行。

[工作流程]
    理解目标：从上下文确认用户要自驱什么，不清楚就问一句。
    填模板：完成条件、验证、约束逐项填实，填不出的占位行删掉、套话删掉，整条压进 4000 字符。
    给用户：把写好的 /goal 整段放进代码块，附一句"复制这段发送即可启动 /goal"。
    提醒：涉及 commit、push、部署、删除等不可逆操作的，让用户发送前再扫一眼。

[初始化]
    读当前上下文，理解目标后按模板生成。

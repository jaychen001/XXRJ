[定位]
    这是追问纪律的约束工具，不是问卷脚本。默认走创造性追问，想接受敷衍时翻 [不接受的答案]，想提前停时翻 [接受标准]。
    不暴露 Phase 标签和维度名给用户。

[五段结构]
    覆盖意图 = 为什么问。主问题 = 默认抛的问题。追问深化 = 用户敷衍时的反击，格式"用户说 X → 你回 Y"。接受标准 = 通过条件。不接受的答案 = 必须驳回的样本。

[Phase 1 · 业务、形态与受众]
    先做，不准跳视觉。

    [1.1 产品形态]
        [覆盖意图]
            形态决定设计的本体和可用的视觉轴，是整份 Brief 的分轨开关。界面型走页面与组件，终端型走字符渲染，对话式 Agent 走会话流与交互范式，嵌入式扩展寄生在宿主里。认错形态，后面每个问题都问错。
        [主问题]
            "这产品长在哪、用户怎么跟它交互？有界面的网页或 app、跑在终端的命令行、像对话一样你来我往的 Agent、还是嵌在别的工具里的扩展？可以叠加，命令行 Agent 就既是终端又是对话式。"
        [追问深化]
            用户说"做个像那种终端编程助手的东西" → 你回："那是终端 + 对话式 Agent + 多 surface 三合一，它没有传统页面，有的是会话流、工具调用、授权确认。后面别问它'侧边导航放哪'，那是问错形态。"
            用户只说"做个 app" → 你回："界面型 app 走页面和组件这套。先确认它不是命令行、不是对话式 Agent，这三套设计本体不一样。"
        [接受标准]
            形态明确，知道下游走界面型本体还是终端 / Agent 本体；叠加形态各占哪块点清。
        [不接受的答案]
            默认当成网页 app 就往下走；说不清用户怎么和它交互。

    [1.2 目标与产品类型]
        [覆盖意图]
            视觉服务于业务，先知道这设计要达成什么、是什么类型，类型直接定信息密度和字号系统。
        [主问题]
            "这产品的单一目标是什么？用户看第一眼该被带去做什么？界面型再分内容阅读、数据仪表盘、交易流程、营销站；终端型和对话式 Agent 分一次性命令工具还是长时自驱 Agent，后者要重点设计运行态和授权。"
        [追问深化]
            用户一上来就聊颜色 → 你回："先别聊颜色，颜色是最后挣来的。先告诉我这东西给谁用、要他干什么。"
            用户说"就是个 app" → 你回："内容型和仪表盘型是两套设计。你这屏给人慢慢读，还是塞满数据高效扫？"
        [接受标准]
            单一目标 + 产品类型明确，定了密度方向。
        [不接受的答案]
            一上来就定颜色风格；说不出产品类型。

    [1.3 受众]
        [覆盖意图]
            受众决定视觉语言、信息密度、节奏。
        [主问题]
            "谁用？描述你理想用户的一天，他什么时候、在什么设备上打开这东西？"
        [追问深化]
            用户说"所有人" → 你回："所有人等于没有人。挑一个主受众，他的审美预期是什么？开发者偏深色密集，消费者偏亲切引导。"
        [接受标准]
            具体受众画像 + 他的审美预期方向。
        [不接受的答案]
            "所有人"；说不出受众的视觉预期。

[Phase 2 · 情绪与人格]

    [2.1 情绪关键词]
        [覆盖意图]
            拿到 3 到 5 个不矛盾的形容词，作为后面每个视觉决策的标尺。
        [主问题]
            "挑 3 到 5 个词描述这产品该给人什么感觉，可信、利落、活泼、高级、冷静、技术感都行。超过 5 个我会逼你砍，词一多就互相打架。"
        [追问深化]
            用户给了 8 个词 → 你回："砍到 5 个以内。'活泼'和'沉稳'同时要等于都没要，留哪个？"
            用户说"高级感" → 你回："高级感是哪种？苹果那种大留白安静克制，还是爱马仕那种深色配金？产品是个走进房间的人，穿西装还是卫衣？"
        [接受标准]
            不超过 5 个形容词，彼此不冲突，每个能映射到操作性的 do 和 don't。
        [不接受的答案]
            8 到 12 个词的愿望清单；纯抽象不带画面；互相矛盾又"都要"。

    [2.2 品牌人格与第一印象]
        [覆盖意图]
            用人格和一个词锁定整体气质。
        [主问题]
            "如果这产品是个人，他穿什么、在哪种餐厅吃饭？还有，用户落地 3 秒内，你最想他得到的那一个感觉是什么，一个词。"
        [追问深化]
            用户说"专业" → 你回："专业有很多种。TED 讲者那种温和权威、Apple Keynote 那种神圣感、还是麦肯锡那种数据冷静？"
        [接受标准]
            一个具体人格画像 + 一个第一印象词。
        [不接受的答案]
            "专业、高大上"这类结果词当人格。

    [2.3 设计原则]
        [覆盖意图]
            从情绪和方向导出 3 条可操作的设计原则，作为后面每个决策的判据。
        [主问题]
            "把这产品的设计信条写成 3 条可操作的原则，不是'好看'这种空话，而是'信息密度优先于装饰'这种能指导取舍的。"
        [接受标准]
            3 条可操作、能指导取舍的原则。
        [不接受的答案]
            '好看、现代'这类空话当原则。

[Phase 3 · 参考与反参考]
    搜索增强双遍在这发力。

    [3.1 喜欢的参考，form 与 content 分开]
        [覆盖意图]
            拿到喜欢的真实产品 + 每个的具体为什么，把"像 Stripe"当症状不当规格。
        [主问题]
            "说 3 到 5 个你欣赏其外观的产品，任何行业都行，每个告诉我具体喜欢哪一点。"
        [追问深化]
            用户说"做得像 Stripe、Linear、Notion" → 你回："你想到它时，真正打动你的那一个东西是什么，克制留白、信息密度、渐变动效、还是把复杂讲简单？把它的颜色和动画全剥掉，还有什么核心感觉必须保留？"
            用户照搬 Apple 大留白营销感套到信息密集工具上 → 你回："你的内容是密集数据，受众要效率。密度该由受众和内容决定，不是只看品味。要 Linear 那种密集仪表盘，还是 Apple 那种留白营销感？"
            终端型或对话式 Agent 的用户拿界面产品当锚 → 你回："那是界面型的壳，你要做的是终端 / 对话式 Agent，结构对不上。借它的气质可以，借它的页面布局不行。找一个同形态的，命令行工具或编程 Agent，说说它哪好。"
        [接受标准]
            正向参考有具体产品 + 每个有为什么；命名产品已拆成属性，分清了哪些是 load-bearing；密度由受众内容决定；锚点同形态，跨形态只借气质不借结构。
        [不接受的答案]
            "像 Stripe"被当字面规格照抄；只抄外壳不懂背后策略。

    [3.2 反参考]
        [覆盖意图]
            反例是边界 fence，用户没态度就会按平均审美自由发挥，做出标准但平庸。
        [主问题]
            "说 2 到 3 个你觉得不对、觉得 low 的产品，具体哪里不对。"
        [追问深化]
            用户说"不要太花哨" → 你回："花哨等于什么？粒子？转场密？颜色多？给一个具体反例。"
            用户说"没什么不想要的" → 你回："那你就是没态度。强制找一个，你刷到什么样的界面会立刻关掉？"
        [接受标准]
            至少 1 个反例 + 至少 3 条绝对不要。
        [不接受的答案]
            "都行、没什么不要的"。

[Phase 4 · 具体 token]

    [4.1 模糊词逼成属性]
        [覆盖意图]
            把人人理解不同的抽象词锚成一个例子、一个滑块位置、或一个命名属性。映射见 SKILL 的 [感受翻译表]。
        [主问题]
            听到 clean、premium、modern、pop 这类词，按感受翻译表逼具体，每个 recap 回去确认。
        [追问深化]
            用户说"要简洁" → 你回："你说的简洁是东西更少、留白更多、颜色更少、还是字体更简单？给我看一个你觉得 clean 的站和一个你觉得乱的站，我好知道你的线在哪。"
            用户说"让它 pop" → 你回："指给我到底哪个元素要跳出来，要更亮、更大、还是周围更多留白？"
        [接受标准]
            每个抽象词落成了一个例子、一个滑块值、或一个命名属性，且 recap 用户点头。
        [不接受的答案]
            形容词光秃秃没锚；"说不清就那种感觉"还不肯指例子。

    [4.2 二选一定轴，禁中间值]
        [覆盖意图]
            每个视觉轴用闭合二三选项砍掉一整枝歧义，滑块禁中间值逼方向。
        [主问题]
            先问现成资产："有没有现成 logo、品牌色、VI 要沿用？有就以它为准，色彩围着它转；没有才从零选。"
            然后每轴一个，给真实产品当锚点：
            "信息密度：comfortable 大量呼吸感像消费 app，还是 compact 一屏更多数据像仪表盘？高级用户要不要能切换？"
            "色彩模式：浅色、深色、还是都要？都要的话哪个默认，深色是一等公民还是锦上添花？"
            "色彩温度：冷色科技克制像 Stripe、Linear，暖色亲和有人味像 Airbnb、Notion，还是中性？"
            "排版气质：几何无衬线利落像 Inter，人文衬线有阅读感像 Newsreader，还是机械等宽技术感？标题要醒目还是克制？"
            "个性：1 是大胆有观点、5 是安全让内容说话，落几？"
            "圆角：锐利 0 到 2px 严肃技术、柔和 8 到 12px 友好现代、还是 pill 活泼？挑 vibe，具体值我定。"
            "动效：snappy 快而隐形、有存在感的表现性、还是完全不要？"
            形态分轨：终端型和对话式 Agent 把圆角、阴影、设备断点标 N/A，排版气质坍缩成等宽，只问纯等宽还是近等宽带变宽标题。换上三条终端专属轴——渲染基元 ASCII 朴素还是 box-drawing 精致；色彩深度 ANSI 16 色保守兼容、256 色、还是 true-color 鲜明；横向收窄按列宽 degrade，80 列窄屏怎么塌。
        [追问深化]
            用户选滑块中间 3 → 你回："不许选中间。偏哪边？选一边能省掉一堆下游扯皮。"
        [接受标准]
            现成品牌资产已确认有无；每轴一个明确方向值，覆盖色彩模式、色彩温度、排版气质、密度、个性、圆角、动效；深色和密度作为主题决策早定；可访问性 WCAG AA 和 reduced-motion 已表态。
            终端 / Agent 形态改覆盖渲染基元、色彩深度、列宽 degrade，圆角、阴影、设备断点标 N/A。
        [不接受的答案]
            没确认有无现成品牌资产就开始选色；哪轴都选中间；深色或 compact 当后期皮肤；可访问性留空。

[Phase 5 · 核心呈现与状态]
    形态分轨：界面型走 5.1 5.2 页面与状态；终端型和对话式 Agent 走 5.3 到 5.8 的 Agent 本体。叠加形态两轨都采，与本形态无关的跳过。

    [5.1 核心页面视觉]
        [覆盖意图]
            Spec 里每个有设计决策空间的页面确认视觉方向，纯后端跳过。
        [主问题]
            "这个页面核心交互是什么？视觉上想要什么方向，给个参考产品。"
        [追问深化]
            用户说"跟整体一样就行" → 你回："这页有它特有的东西吗？列表页要不要密、详情页要不要留白，说一个。"
        [接受标准]
            每个核心页面有视觉方向 + 参考。
        [不接受的答案]
            每页都"跟整体一样"不给任何页面特性。

    [5.2 状态设计]
        [覆盖意图]
            空、加载、错误三态的视觉方向，不甩给设计开发瞎猜。
        [主问题]
            "空状态用插画引导还是极简文案？加载用骨架屏、转圈、还是打字机？错误是内联提示还是弹窗？"
        [接受标准]
            三态各有方向。
        [不接受的答案]
            三态留空。

    [5.3 交互骨架与模式]
        [覆盖意图]
            终端和对话式 Agent 没有页面导航，骨架是用户怎么发起、Agent 怎么回、有哪些模式和命令，这是界面型信息架构的对应物。
        [主问题]
            "用户怎么驱动它？纯自然语言、还是有 slash 命令和快捷键？有没有模式切换，先看计划再执行的计划模式、只读模式、自动执行模式？一轮交互长什么样，从用户发起到 Agent 收尾？"
        [追问深化]
            用户说"就是个聊天框" → 你回："纯聊天框对 Agent 不够。它要不要命令、要不要模式、长任务要不要能切后台？光一个输入框，复杂操作全靠打字描述，体验会很糙。"
        [接受标准]
            交互入口 + 命令与快捷键方向 + 模式清单 + 一轮交互骨架说清。
        [不接受的答案]
            只有"一个聊天框"，不谈命令、模式、长任务控制。

    [5.4 呈现单元与技术内容]
        [覆盖意图]
            Agent 的"页面"是一条条呈现单元：消息轮、工具调用、diff、代码、思考、计划、待办。技术内容在终端或对话里怎么呈现，是这类产品的核心视觉决策。
        [主问题]
            "Agent 的输出由哪些单元组成，用户消息、Agent 回复、工具调用、代码 diff、命令输出、思考过程、计划、待办？每种怎么呈现，代码要不要高亮，diff 要不要分行着色，长输出要不要折叠？"
        [追问深化]
            用户说"输出就是文字" → 你回："Agent 跑命令、改文件、报错，塞成一坨纯文字没法看。工具调用要不要单独成块、diff 要不要绿红、报错要不要和正常输出区分？指一个你觉得读着舒服的终端工具。"
        [接受标准]
            呈现单元清单 + 每种呈现方向 + 代码 / diff / 错误 / 长输出的处理方向。
        [不接受的答案]
            所有输出一种样式；技术内容不分层。

    [5.5 Agent 运行态]
        [覆盖意图]
            界面型六态不够。Agent 有自己的态：流式输出中、思考中、跑工具中、等授权、被中断。每个态要让用户一眼知道现在什么情况。
        [主问题]
            "Agent 干活时怎么让用户知道在干嘛？流式打字、思考中、正在跑某个工具、等你点头授权、被你打断，这几个态分别怎么显示？长任务要不要进度或实时日志？"
        [追问深化]
            用户说"转个圈就行" → 你回："一个圈用户分不清它卡住还是在干活。正在读文件、正在改代码、正在跑测试，分不分得清？长任务全程一个圈，用户会以为死了。"
        [接受标准]
            流式 / 思考 / 跑工具 / 等授权 / 被中断各有显示方向；长任务有进度方向。
        [不接受的答案]
            所有运行态共用一个 spinner；长任务无进度。

    [5.6 透明度与信任]
        [覆盖意图]
            Agent 替用户动手，用户凭什么信。把在做什么、为什么这么做、做过什么露到什么程度，是信任的设计决策。
        [主问题]
            "Agent 的思考和理由露多少？全程展示推理、只给结论、还是可展开？做过的动作留不留痕迹给用户回溯？露太多吵，露太少黑箱，线划在哪？"
        [追问深化]
            用户说"用户不用看过程" → 你回："Agent 删了文件、改了配置，用户事后想知道它干了啥都没处查，出事就炸锅。至少留个可回溯的动作记录，默不默认展开是另一回事。"
        [接受标准]
            推理展示程度 + 动作可回溯方向定了。
        [不接受的答案]
            Agent 自动动手却不留痕、不可回溯。

    [5.7 授权与同意]
        [覆盖意图]
            Agent 能跑命令、改文件、花钱，哪些动作要先问、问的粒度多细、危险动作怎么拦，是这类产品最关键的控制设计。
        [主问题]
            "哪些动作 Agent 能直接做，哪些要先问用户？授权粒度多细，每次都问、本会话记住、还是永久放行？删文件、推代码、花钱这类危险动作怎么拦，要不要二次确认？"
        [追问深化]
            用户说"都让它自己干" → 你回："全自动跑到删库或者把密钥提交上去，谁兜底？危险动作至少分级，哪些必须先问、哪些可以本会话放行，给个边界。"
        [接受标准]
            动作分级 + 授权粒度 + 危险动作拦截方向。
        [不接受的答案]
            全自动无任何 gate；危险动作不分级。

    [5.8 多 surface 一致性]
        [覆盖意图]
            同一个 Agent 常有多个壳：终端、编辑器扩展、桌面、网页。一套设计要在不同壳里都成立，哪些一致、哪些因壳而变，得早定。
        [主问题]
            "这 Agent 会有几个壳，终端、IDE 扩展、桌面、网页？哪些是一等公民、哪个先做？跨壳什么必须一致，什么可以因壳制宜，比如终端用字符、桌面能用真图形？"
        [追问深化]
            用户说"先做终端，以后再说" → 你回："那现在的设计决策别堵死多壳路。哪些是终端特有、换到图形界面就不成立的，先标出来，免得以后返工。"
        [接受标准]
            壳清单 + 优先级 + 跨壳一致与可变的边界。单壳产品明确标单壳，跳过。
        [不接受的答案]
            多壳却假设只有终端；不分一致与可变。

[Phase 6 · 结构与落地规格]
    分级采集：方向和清单访谈采到，token 具体值、文案细则标 [待补充] 交设计 / dev。喂 template 的信息架构、页面规格、组件规格、UX 文案。
    形态分轨：界面型走 6.1 到 6.5。终端型和对话式 Agent 的信息架构就是 5.3 交互骨架，页面规格就是 5.4 呈现单元加 5.5 运行态，响应式走列宽 degrade 加 5.8 多 surface，另必落 5.6 5.7 透明度与授权；6.3 组件和 6.4 文案两形态都采，组件锚点按形态换。

    [6.1 信息架构与导航]
        [覆盖意图]
            定导航结构和页面清单，每页关联 Spec 的流程 / 功能，这是设计的骨架。
        [主问题]
            "产品用什么导航（顶部 / 侧边 / 底部 Tab / 向导式）？有哪些页面，每页对应 Spec 的哪个 FLOW / REQ？页面之间怎么跳？"
        [接受标准]
            导航结构 + 页面清单（每页关联 FLOW / REQ + 优先级）+ 页面关系。
        [不接受的答案]
            没有页面清单；页面不关联 Spec。

    [6.2 页面规格]
        [覆盖意图]
            每个有设计决策空间的页面，定布局、内容层级、用的组件、六态、响应式。
        [主问题]
            "这页布局怎么分区？哪些信息 / 操作最重要（内容层级）？用哪些组件？默认 / 加载 / 空 / 错误 / 成功 / 无权限六态分别什么样？移动和桌面怎么变？"
        [追问深化]
            用户说"常规布局" → 你回："最重要的操作放哪、第一眼看到什么？这页和列表页 / 详情页有什么不同？"
            用户在做剪辑、设计、数据分析、编码等工作台型产品 → 你回："先标出不可压缩的主任务区域：画布、预览、时间线、编辑区或结果表。导航、Agent、属性、日志这些辅助功能放哪，不能挤压主操作空间。能用既有侧栏、Tab、抽屉或上下文面板解决，就别默认加第二侧栏。"
        [接受标准]
            每个核心页面有布局 + 内容层级 + 组件 + 六态 + 响应式方向；工作台型页面明确主任务区域和辅助区承载方式；纯后端页跳过。
        [不接受的答案]
            页面无布局和层级；六态留空；工作台型页面默认增加会挤压主任务区域的第二侧栏。

    [6.3 组件规格]
        [覆盖意图]
            列核心可复用组件，定变体和状态方向，避免一个按钮画十遍。
        [主问题]
            "有哪些重复出现的核心组件（按钮、输入框、卡片、导航项…）？每个有哪些变体（主 / 次 / 危险）？状态（默认 / 悬停 / 聚焦 / 禁用 / 加载 / 错误）方向？"
        [接受标准]
            核心组件清单 + 每个变体 + 状态方向；具体样式值标 [待补充] 交设计工具。
        [不接受的答案]
            不列组件，页面各画各的。

    [6.4 UX 文案]
        [覆盖意图]
            定文案语气和按钮 / 表单 / 空态的文案规则方向，统一产品声音。
        [主问题]
            "界面文案什么语气（清晰 / 友好 / 专业 / 鼓励）？按钮文案怎么写（动词开头？）？表单 label / placeholder / error 规则？空状态要不要解释原因加给下一步？"
        [接受标准]
            文案语气 + 按钮 / 表单 / 空态文案规则方向；具体文案标 [待补充]。
        [不接受的答案]
            文案语气不定；空态无引导。

    [6.5 响应式]
        [覆盖意图]
            移动 / 平板 / 桌面各档的布局规则，不只桌面。
        [主问题]
            "移动端布局怎么变（单列？折叠？）？平板要不要单独处理？桌面端最宽多少？375px 会不会横向滚动？"
        [接受标准]
            移动和桌面至少两档有布局规则；关键断点定了。
        [不接受的答案]
            只考虑桌面；移动端布局留空。

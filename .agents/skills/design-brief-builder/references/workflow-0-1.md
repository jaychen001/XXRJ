---

[使用时机]
    有 Product-Spec.md，没有 Design-Brief.md，用户要定视觉方向。

[顶层规则]
    Phase 顺序：先业务受众，最后才视觉。别先问颜色风格，视觉方向是挣来的。
    每个 Phase 达到出口判据才解锁下一个。
    形容词永远不算合格答案，按 SKILL 的 [感受翻译表] 和 references/interview-principles.md 的四把刀逼具体。
    每相位结束默扫 interview-principles 的反失败自检。
    Phase 标签不暴露给用户，用承上启下的人话过渡。
    详细问法见 references/question-bank.md 对应 Phase。
    分级采集：方向和清单（情绪、参考、原则、信息架构、页面、组件、文案语气）问到底；token 具体值、文案细则标 [待补充] 交设计工具和 dev。
    编号纪律：页面 SCREEN、组件 CMP 编号，关联 Product Spec 的 FLOW / REQ；两份文档的 P0 内容保持一致。终端 / Agent 形态没有 SCREEN，落到 template §A 的交互骨架、呈现单元、运行态，同样关联 FLOW / REQ。

[Phase 0 · 启动]
    搜索增强第一遍：提问前 WebSearch 用户行业竞品 + 跨行业标杆 + 当前趋势，带具体参考进场。
    加载 Product-Spec.md，从产品类型、目标用户、功能提取设计采访的基础，不重复问。

[Phase 1 · 业务、形态与受众，先做]
    覆盖：产品形态、单一目标、具体受众、产品类型。形态定走界面型还是终端 / Agent 本体，类型定信息密度和字号系统。
    出口判据：形态明确，能说清这设计为谁、达成什么、是哪类产品。

[Phase 2 · 情绪与人格]
    覆盖：3 到 5 个不矛盾形容词、品牌人格、第一印象一个词。
    出口判据：形容词收敛到 5 个内且不互斥。

[Phase 3 · 参考与反参考]
    覆盖：3 到 5 个喜欢的产品加为什么，2 到 3 个反参考加哪里不对。
    搜索增强第二遍在这里：用户每报一个风格词或产品名，立刻当 query 再搜 2 到 4 个同调性参考，带回做二选一收窄，拆成间距、色彩、字体、动效、语气五轴。
    出口判据：正反参考各有具体产品 + 拆成了属性。

[Phase 4 · 具体 token]
    覆盖：模糊词按感受翻译表逼成属性；每个视觉轴二选一定方向，禁中间值；可访问性表态。
    出口判据：色彩、密度、排版、圆角、动效各有方向，深色和密度作为主题决策定了。

[Phase 5 · 核心呈现与状态]
    形态分轨：界面型采页面视觉 + 三态；终端型和对话式 Agent 采交互骨架、呈现单元、运行态、透明度、授权、多 surface，详见 references/question-bank.md 的 5.3 到 5.8。

[Phase 6 · 结构与落地规格]
    覆盖：信息架构与导航（页面清单关联 FLOW / REQ）、页面规格（布局 / 内容层级 / 组件 / 六态 / 响应式）、组件规格（变体 / 状态方向）、UX 文案语气与规则、响应式断点。token 具体值和文案细则按分级标 [待补充]。

[完成度判据]
    交接测试：把 Brief 交给设计工具或没参与对话的设计师，能照着出稿不用猜吗？要猜的地方就是没够。
    相位级：能复述该相位的决定，用户点头，就进下一个。
    收尾：把解码出的方向当可观察模式 recap-and-confirm，"所以是高对比强调色 + 大留白 + 无衬线 + 克制功能性动效，对吧"，作为视觉契约。加覆盖清单 + 元问题。达标才生成。

[生成]
    生成前先把感受翻译成设计属性，检查一致性，矛盾逼取舍。
    按 templates/design-brief-template.md 生成 Design-Brief.md，每个方向附参考产品；页面 SCREEN、组件 CMP 编号，关联 Product Spec 的 FLOW / REQ。
    没定的标 [待补充]，不凭空补；token 具体值交设计工具。
    确保和 Product Spec 的 P0 内容一致。
    生成后引导下一步：喂给设计工具画设计稿，或调用 /dev-planner。

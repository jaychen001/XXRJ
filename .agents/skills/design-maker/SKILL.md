[任务]
    读 Product-Spec.md 和 Design-Brief.md，通过设计工具 MCP 生成完整设计交付物。Spec 里每个有 UI 的功能都要有设计页面，每个页面覆盖所有关键状态变体。

[依赖检测]
    必需：Product-Spec.md、Design-Brief.md、设计工具 MCP。
    设计工具检测：问用户用 Pencil 还是 Figma → 检测 MCP 是否连接 → 没连尝试连或提示用户 → 用户跳过则退出，后续按无设计稿模式继续。

[文件结构]
    design-maker/
    └── SKILL.md  # 本文件，无 references / templates

[第一性原则]
    完整覆盖：Spec 里每个有 UI 的功能都有设计页面。漏一个，开发就少一个参照，只能靠猜。
    状态完备：每个页面不只默认态。有交互的覆盖空状态、加载态、错误态、激活态。
    组件先行：先建可复用组件再拼页面，避免一个按钮画 10 遍、改一次改 10 处。
    文档驱动：一切设计决策来自 Spec 和 Brief，不凭个人偏好发挥，不加文档没写的功能。

[设计交付物]
    设计变量：从 Brief 提取颜色、字体、间距、圆角系统，设到设计工具。
    可复用组件：按钮、输入框、导航项、卡片、标签等重复元素。
    所有页面：从 Spec 的 UI 布局、功能需求、用户流程三处交叉提取页面清单，用组件拼装，布局间距对照 Spec。
    状态变体：默认态所有页面都有；有数据展示的加空状态，有异步的加加载态，有可能失败的加错误态。

[工作流程]
    加载 Spec 和 Brief → 提取页面清单、状态变体、组件清单 → 向用户展示设计交付清单和总数，确认后开始 → 取设计工具 guidelines → 设变量 → 建组件逐个截图验证 → 逐页面设计，用组件拼、填真实内容不用 Lorem ipsum、对照 Spec 和 Brief、截图验证 → 设状态变体 → 完整性和一致性校验，对照 Spec 确认无遗漏 → 输出报告引导 /dev-planner。

[初始化]
    执行加载和规划。

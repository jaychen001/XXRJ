---
name: question-bank
description: product-spec 需求采集的问题库，按 Phase 组织。每个维度含覆盖意图、主问题、追问深化、接受标准、不接受的答案。访谈时翻它校准，不是顺序问卷。
---

[定位]
    这是追问纪律的约束工具，不是问卷脚本。默认走创造性追问，想接受敷衍时翻 [不接受的答案]，想提前停时翻 [接受标准]，想瞎问时翻 [主问题]。
    不暴露 Phase 标签和维度名给用户，那是你内部翻题库用的。

[五段结构]
    覆盖意图 = 为什么问这维度。主问题 = 默认抛的问题。追问深化 = 用户敷衍时的反击，格式"用户说 X → 你回 Y"。接受标准 = 通过条件。不接受的答案 = 必须驳回的样本。

[Phase 1 · 问题与人]
    地基，必须过。

    [1.1 真实问题与唯一用户]
        [覆盖意图]
            锁定一个具体主用户和一个被真实经历过的问题，避免为所有人解决一切。
        [主问题]
            "这产品解决的那一个问题是什么？谁最痛？描述一个上周真撞上这问题的具体人。"
        [追问深化]
            用户说"所有人都会用" → 你回："所有人等于没有人。第一版只为一种人做，是谁？现在明确不为谁做？"
            用户说"大家都觉得 X 烦" → 你回："别人我不管。你上次被它烦到是什么时候，当时具体在干嘛，一步步讲。"
            用户说"现在没人解决这个" → 你回："那他们今天怎么凑合的？拿什么工具、表格、土办法顶着？答案真是'什么都没有'，这问题可能根本不痛。"
            用户给的是观点不是故事 → 你回："这是你的判断。给我一个真实发生过的例子。"
        [接受标准]
            一个可指认的主用户 + 一个有时间地点的真实问题实例 + 现状 workaround 说清。
        [不接受的答案]
            "所有人、大家、用户、一个市场"；纯假设无实例；赞美；描述的是你的方案不是他的处境。

    [1.2 现状与代价]
        [覆盖意图]
            用过去行为验证问题是真的，量化痛度，分清止痛药和维生素。
        [主问题]
            "带我走一遍，你今天怎么处理这件事，从头到尾，用了哪些工具、哪些步骤？"
        [追问深化]
            用户说"我一般都、我总是、我会" → 你回："别说一般。上次这么干是什么时候？那一次具体发生了什么，走一遍。"
            用户说"挺花时间的" → 你回："上次花了多久？一周来几次？折算一个月多少时间、多少钱？"
            用户说"我会很愿意付费" → 你回："你已经为解决它花过钱吗？买过什么工具、拼过什么方案？花了多少？没有的话为什么没有？"
            用户说现状方案"不好用" → 你回："具体哪一步崩的？上次它坑你是怎么坑的？"
        [接受标准]
            有过去真实行为 + 代价被量化成数字 + 有已付出的努力或金钱作为真实需求证据。
        [不接受的答案]
            只有未来意愿无过去行为；"会付费"当验证；模糊的"很烦、很慢"不带数字。

[Phase 2 · Job 与成功]
    地基，必须过。

    [2.1 无方案的 Job 陈述 + 成功度量]
        [覆盖意图]
            拿到一个稳定、不含任何产品功能的 job 陈述，和一个可度量的期望结果，作为整个 Spec 的锚。
        [主问题]
            "把核心任务写成一句不提任何 app、任何功能的话，帮某类人[动词][对象]，当[情境]时。这个动词是什么？"
        [追问深化]
            用户答里夹了方案"用我的 app 来…" → 你回："去掉'用 app'。job 是稳定的，产品是会变的。不提工具，他到底要完成什么？"
            用户说"让它更简单" → 你回："'更简单'不是需求。他想最小化什么、最大化什么？比如把检测到异常的时间降到最短。"
            用户给不出成功度量 → 你回："这产品完美运转后，今天不成立的什么会变成立？他怎么知道自己成功了？给我一个能数出来的信号。"
            用户列了一堆指标 → 你回："挑一个北极星，和用户 job 直接挂钩的那一个。别挑注册数这种好刷又虚的。"
        [接受标准]
            job = 动词 + 对象 + 情境，零方案；至少一个带度量的期望结果；一个不可灌水的成功度量。
        [不接受的答案]
            job 里烘焙了解法；"更好、更易用"当需求；北极星是注册量下载量这类虚荣指标；成功条件无法度量。

[Phase 3 · 范围与非目标]
    早做，必须过。

    [3.1 范围与非目标]
        [覆盖意图]
            在开发前显式钉死不做什么，砍掉模糊野心。
        [主问题]
            "这一版明确不做什么？给我至少 3 条故意选择不碰的东西。"
        [追问深化]
            用户说"先都做了再说" → 你回："那就永远发不出来。只能上线一个功能，是哪个？其余你愿意先不要的是什么？"
            用户冒出新点子 → 你回："这点子不错，我先记到'以后'清单别丢，咱们先把核心钉死，行吗？"
            用户说不出非目标 → 你回："如果这产品明天消失，用户会改用什么？他会真的难受吗？"
        [接受标准]
            至少 3 条显式非目标 + v1 的单一必须功能 + 维生素还是止痛药判断清楚。
        [不接受的答案]
            "都要做"；没有任何不做清单；说不出 v1 最小集。

    [3.2 优先级分级]
        [覆盖意图]
            v1 内部也分轻重，P0 没有就不能发，P1 有余力才做，避免什么都 P0。
        [主问题]
            "本版本要做的里，哪些是没有就不能上线的 P0？哪些是 P0 做完有余力才做的 P1？"
        [追问深化]
            用户全标 P0 → 你回："全 P0 等于没优先级。只能上线一个，留哪个？其余降 P1。"
        [接受标准]
            每个范围项有 P0/P1；P0 是发布的最小集。
        [不接受的答案]
            全 P0；说不出哪个能砍到 P1。

[Phase 4 · 旅程与功能]

    [4.1 端到端用户旅程]
        [覆盖意图]
            把用户从打开到完成任务的完整路径走出来，暴露缺失的步骤和分支。
        [主问题]
            "带我从用户打开产品的第一秒，走到他完成任务的那一刻，一步一步，每步他做什么、看到什么。"
        [追问深化]
            用户跳步"然后他就完成了" → 你回："中间那步呢？他怎么从 A 到 B 的，点了什么、填了什么？"
            用户只讲理想路径 → 你回："如果这一步他填错了、或者没数据，会怎么样？"
        [接受标准]
            至少一条完整路径从头到尾无断点；主要分支点已标出。
        [不接受的答案]
            跳步；只有理想路径无异常分支。

    [4.2 功能拆解]
        [覆盖意图]
            每个核心功能拆到"用户做什么 → 系统做什么 → 得到什么"，消灭"实现 XX 功能"这种空壳。
        [主问题]
            "这个功能，用户做什么动作触发，系统做什么，他最后得到什么？"
        [追问深化]
            用户说"实现一个推荐功能" → 你回："用户在哪触发？系统拿什么数据算？他看到几条、怎么排、能不能调？"
            用户把多个功能糊一起 → 你回："这是几个功能，拆开，每个单独说触发和产出。"
        [接受标准]
            每个核心功能有触发 + 系统行为 + 用户可感知产出；核心和辅助功能分清。
        [不接受的答案]
            "实现 XX"没有三段；功能糊作一团。

    [4.3 用户任务]
        [覆盖意图]
            列出用户要完成的任务，独立于功能怎么实现，任务是流程和功能的锚，喂 template 的 TASK 表。
        [主问题]
            "用户用这产品要完成哪几件事？每件用'用户[动词][对象]'描述，别提具体功能。"
        [追问深化]
            用户拿功能名当任务 → 你回："那是功能。用户的目标是什么？不是'用搜索框'，是'快速找回某条信息'。"
        [接受标准]
            任务清单，每个是用户目标级、带优先级；一个任务可对应多个功能。
        [不接受的答案]
            把功能当任务；任务和功能糊一起。

    [4.4 数据模型]
        [覆盖意图]
            功能要存取数据，定核心实体、关系、数据规则，不让开发猜数据结构，喂 template 的数据模型表。
        [主问题]
            "核心数据对象是什么（用户、订单、文档…）？它们之间什么关系？数据怎么创建/更新/删除，谁能看谁能改？"
        [追问深化]
            用户只说功能不说数据 → 你回："这功能背后存什么？一条记录有哪些字段？"
        [接受标准]
            核心实体 + 关系 + 数据规则（CRUD + 权限）；纯前端无持久化可略。
        [不接受的答案]
            说不清数据结构就开发。

    [4.5 功能规则与输入]
        [覆盖意图]
            功能拆解之上补每个功能的 MUST/SHOULD 规则和输入字段校验，喂 template 的 REQ 规则与输入表。
        [主问题]
            "这功能必须遵守什么（MUST）、最好遵守什么（SHOULD）？用户输入哪些字段，每个什么类型、必填吗、怎么校验？"
        [追问深化]
            用户跳过输入校验 → 你回："这字段空了、超长了、格式不对，怎么办？校验规则写出来。"
        [接受标准]
            每个核心功能有 MUST/SHOULD 规则 + 输入字段的类型和校验。
        [不接受的答案]
            规则模糊；输入字段无校验。

[Phase 5 · AI 能力]

    [5.1 AI 能力选型]
        [覆盖意图]
            确认哪些功能真需要 AI、用哪类能力，先过 litmus test 防止为 AI 而 AI。
        [主问题]
            先跑 litmus，"把 AI 换成写死规则或一个搜索框，核心价值还在吗？"再对照 SKILL 的 [AI 能力对照表] 把菜单摆成多选。
        [追问深化]
            用户说"用 AI 做" → 你回："具体哪类？是帮他生成文字、看懂图、还是基于你的文档回答？对照这几个能力，你要哪个？"
            缺信息的场景 → 你回："答案该从模型的通用知识来，还是从你自己会更新的文档来？后者用 RAG。"
            用户要 agent → 你回："这是要 AI 自己动手跨工具做多步，还是只要它建议、你来按确认？多步自动最贵，先确认值不值。"
        [接受标准]
            每个 AI 功能明确能力类型 + 过了 litmus；选型顺序走过一遍。
        [不接受的答案]
            "用 AI"不指类型；为 AI 而 AI 没过 litmus。

    [5.2 AI 概率性四问，每个 AI 功能必问]
        [覆盖意图]
            AI 是概率性的，质量、触发、失败、成本不钉死，开发就在赌。
        [主问题]
            质量条："给我 5 个'好答案'的例子和 5 个'丢脸答案'的例子。最重要那个质量维度，命中率到多少你敢上线，80% 还是 95%？低于多少要报警？"
            触发方式："只是建议或草稿让用户确认，还是自动执行？"默认建议优先，先建信任。
            失败兜底："这功能能犯的最贵的错是什么？它绝不能做什么哪怕用户要求？AI 不确定时该说'我不确定'、给出处、追问、还是转人工？服务挂了或慢了，禁用提示、降级缓存、还是排队？"
            延迟成本："必须多快，3 到 5 秒可接受还是要瞬时？流式输出行吗？一天大概跑多少次，单次成本有上限吗？"
        [接受标准]
            每个 AI 功能有质量阈值 + 触发方式 + 绝不能做清单 + 不确定时行为 + 降级策略 + 延迟成本预算。
        [不接受的答案]
            "要准一点"无数字；跳过失败兜底；没说触发是建议还是自动。

[Phase 6 · UI 与状态]

    [6.1 布局与控件]
        [覆盖意图]
            把界面结构、各区域内容、主要控件定清楚，不甩给开发猜。
        [主问题]
            "整体是什么布局，几栏、各区域放什么？主要的输入输出控件是什么？"
        [追问深化]
            用户说"界面你看着办" → 你回："布局影响信息怎么组织。给个方向，左边放什么右边放什么，还是上下结构？我可以给 2 到 3 个方案你选。"
        [接受标准]
            整体布局结构 + 各区域内容 + 主要控件清楚。
        [不接受的答案]
            "看着办"不给方向且不肯从方案里选。

    [6.2 五态强制]
        [覆盖意图]
            每个界面的加载、空、错误、成功、无权限五态显式问出来，不让设计开发各自瞎猜。
        [主问题]
            一次一态，别问"边界都处理了吗"这种永答"处理了"的 yes/no：
            "用户第一次打开、还没数据时，空状态显示什么，引导他干嘛？"
            "数据加载时屏幕长什么样，骨架屏、转圈、还是阻塞？几秒算超时？"
            "操作失败时，断网、服务器错、校验不过，具体弹什么文案？他下一步能干嘛，重试、改、还是取消？"
            "谁有权做这个？登出的、没权限的、会话过期的用户点进来分别看到什么？"
        [追问深化]
            用户说"正常显示" → 你回："空结果不能跟报错长一样。这俩分别什么样，差在哪？"
            用户把多个行为捆一起"它校验然后保存然后发邮件" → 你回："这是三个行为，分开说，每个都能单独成功或失败。"
        [接受标准]
            核心界面五态各有明确行为；空态不等于错误态；复合行为已拆分。
        [不接受的答案]
            "处理了、正常显示"；只有理想路径；被动语态"邮件会被发送"。

[Phase 7 · 验收与边界]

    [7.1 验收标准可测化]
        [覆盖意图]
            把关键功能从模糊描述变成可测的 Given-When-Then，消灭"快、直观、无缝"这类无法判定的词。
        [主问题]
            "把这功能主路径写成 Given<状态> When<用户做 X> Then<系统做 Y>。Then 里用户到底看到什么具体的 Y？"
        [追问深化]
            用户写不出 Then → 你回："写不出说明这需求还不成立。用户最后看到的那个终态画面是什么？"
            用户说"要快" → 你回："多快，毫秒数。结果 200ms 内出来？给个数字。"
            用户说"手机上能用" → 你回："多宽的屏？375px 宽时布局怎么变，会不会横向滚动？"
            一个 story 膨胀成迷你 spec → 你回："太大，拆成几条独立可测的。"
        [接受标准]
            关键功能有 GWT 场景，一个场景一个可观察结果；模糊形容词全换成数字或具体断点。
        [不接受的答案]
            留"快、直观、无缝"无数字；复合 GWT 用"然后…然后…"捆多个结果；被动语态藏住谁做什么。

    [7.2 非功能需求]
        [覆盖意图]
            性能、安全、隐私、兼容、可靠、可访问 6 类逐项给数字或明确要求，不一句"要快要安全"，喂 template 的非功能表。
        [主问题]
            "性能多快（响应、并发）？安全和隐私有什么要求（认证、加密、数据合规）？支持哪些设备和浏览器？出错怎么恢复、数据会不会丢？可访问性到什么级别？"
        [追问深化]
            用户说"要快要安全" → 你回："多快，响应毫秒数、并发多少？安全具体指什么，登录、加密、还是权限？给我能验收的。"
            用户说"安全 / 隐私要完善" → 你回："先按产品类型和威胁模型定边界。自用或内部 API 工具默认保护密钥和本地数据，不主动发明逐 provider 数据外传仪表盘；除非合规、发布或用户明确要求。"
        [接受标准]
            6 类各有要求或明确"不要求"，关键项给数字，标 P0/P1；隐私安全要求和产品类型、用户威胁模型、已接受的 API 依赖一致。
        [不接受的答案]
            一句"要快要安全"无数字；6 类全空；给自用或内部 API 工具发明用户不关心的安全仪表盘。

    [7.3 假设与待确认]
        [覆盖意图]
            信息不全是常态，把推断显式记为假设带风险，把不能判断的记为待确认，不自行扩展需求，喂 template 的假设和待确认表。
        [主问题]
            "哪些地方你没明说、我得替你假设？这假设错了会怎样？哪些是你也定不了、要后面确认的？"
        [接受标准]
            关键假设显式记录，带依据和错误风险；待确认问题列出，标是否阻塞。
        [不接受的答案]
            把假设当事实写死；遇到不确定自行扩展需求。

[Phase 8 · Agent 系统专项]
    仅当产品本身是 agent / 自主系统（用户给目标、系统自主执行）时触发，传统界面应用跳过。喂 template 的「Agent 系统规格」。

    [8.1 自主性与人在回路]
        [覆盖意图]
            定 agent 的自主边界——哪些自动、哪些必须人确认，这是 agent 产品第一决策。
        [主问题]
            "agent 能自己做到哪一步？哪些动作必须停下等用户点头（写文件、删除、花钱、对外发消息）？是建议确认、自动可回滚、还是自动不可逆？"
        [追问深化]
            用户说"全自动" → 你回："不可逆的呢？删数据、花钱、对外发东西也自动？哪一步出错最贵，那步就该设审批门。"
        [接受标准]
            每类动作的自主级别（自动 / 确认 / 禁止）划清；危险或不可逆操作有审批门。
        [不接受的答案]
            "全自动"不分动作类别；危险操作无确认。

    [8.2 工具与能力集]
        [覆盖意图]
            agent 靠工具行动，定它有哪些工具、权限边界、扩展机制。
        [主问题]
            "agent 能调哪些工具或动作（读文件、写文件、跑命令、联网、调 API）？每个权限级别？要不要支持扩展（MCP / plugin）？"
        [追问深化]
            用户说"能干所有事" → 你回："工具越多越危险。先给最小集，每个工具配权限。要不要扩展机制让用户加工具？"
            用户要兼容既有 Skill / plugin 标准 → 你回："兼容哪个标准，Codex、Claude Code、MCP 还是别的？包结构支持什么：SKILL.md、frontmatter、scripts、references、assets、examples？脚本和工具调用谁执行，用户怎么通过对话或命令触发？"
        [接受标准]
            工具清单 + 每个工具的权限级别 + 扩展机制有无；若兼容既有标准，必须写清标准来源、包结构、资源类型、脚本 / 工具调用能力、宿主触发执行路径。
        [不接受的答案]
            工具集无边界；权限不分级；把扩展机制简化成纯 Markdown 指令包。

    [8.3 上下文与记忆]
        [覆盖意图]
            上下文管理决定 agent 能处理多大任务、记多久，是 agent 产品的核心工程。
        [主问题]
            "单任务上下文多大够用？超了怎么办（压缩、分段、检索）？跨会话要不要记忆（用户偏好、项目背景）？记在哪、记多久？"
        [追问深化]
            用户没想过上下文 → 你回："任务一长上下文就爆。50 步以上怎么不丢前面的信息？要不要压缩或外部记忆？"
        [接受标准]
            上下文上限策略 + 超限处理 + 跨会话记忆范围。
        [不接受的答案]
            不考虑上下文上限；长任务无管理策略。

    [8.4 编排与多 agent]
        [覆盖意图]
            复杂任务要不要拆给子 agent，怎么协调隔离。
        [主问题]
            "任务复杂时一个 agent 扛到底，还是拆给子 agent 并行 / 委派？子 agent 怎么隔离、结果怎么合并？"
        [追问深化]
            用户说"一个 agent 全做" → 你回："复杂任务一个 agent 容易上下文爆、职责乱。要不要拆专门子 agent？怎么合并它们的结果？"
        [接受标准]
            单 vs 多 agent 决策；多 agent 有隔离和结果合并方式。
        [不接受的答案]
            复杂任务硬塞一个 agent 且无理由。

    [8.5 评估与可观测]
        [覆盖意图]
            agent 是概率系统，没 eval 就是盲飞，这是 agent 产品最容易漏的命脉 primitive。
        [主问题]
            "怎么判断 agent 这次做得好不好？有没有可重跑的评估集 / 回归？线上怎么追每一步（tracing）、出问题怎么定位？质量掉了怎么发现？"
        [追问深化]
            用户没想过 eval → 你回："agent 不是确定性的，今天对明天可能错。你拿什么自动判断它做得对不对、退化了没？没有 eval 就是闭眼上线。"
        [接受标准]
            有评估方式（评估集 / 指标 / 回归）+ 可观测（每步可追）+ 质量退化能发现。
        [不接受的答案]
            没有任何 eval / 可观测；只靠人工偶尔看。

    [8.6 成本与预算]
        [覆盖意图]
            agent 烧 token，成本是产品级约束，不控会失控。
        [主问题]
            "单任务的 token / 成本上限是多少？怎么防失控（cap / 熔断）？要不要按任务选模型（贵任务强模型、便宜任务小模型）？"
        [追问深化]
            用户没想过成本 → 你回："agent 一跑就烧 token，一个 bug 能跑出五位数账单。单次上限多少？超了熔断吗？"
        [接受标准]
            成本上限 + 防 runaway 机制 + 模型路由策略有无。
        [不接受的答案]
            无成本上限；不防失控。

    [8.7 失败与卡死]
        [覆盖意图]
            agent 会卡、会循环、会失败，要有检测和兜底。
        [主问题]
            "agent 卡住 / 循环 / 反复失败时怎么办（检测 + 停 + 报告）？任务失败的中间产物怎么处理（回滚 / 保留 / 继续）？什么情况交回人？"
        [追问深化]
            用户说"它会自己解决" → 你回："它会死循环、会反复试同一个错招。卡几次该停？停了中间产物留还是回滚？什么时候必须交回人？"
        [接受标准]
            卡死 / 循环检测 + 失败兜底 + 交还人的明确条件。
        [不接受的答案]
            假设 agent 永远能自己解决；无卡死检测。

    [8.8 会话与状态]
        [覆盖意图]
            长任务要能中断、恢复，历史要可追溯。
        [主问题]
            "任务能中断后恢复吗（resume）？历史 / transcript 留不留、给谁看？"
        [接受标准]
            会话持久化 + resume + 历史策略。
        [不接受的答案]
            长任务不能中断恢复；无历史。

name: workflow-0-1
description: product-spec 0-1 模式工作流。Phase 划分，从用户初次表达到生成 Product-Spec.md。
---

[使用时机]
    项目里没有 Product-Spec.md，用户第一次表达想做产品。

[顶层规则]
    每个 Phase 达到出口判据才解锁下一个，Phase 1 到 3 是地基。
    允许渐进式：用户初始描述里已覆盖的维度直接吸收并复述确认，不机械重问。
    形容词永远不算合格答案，逼出具体见 references/interview-principles.md 的四把刀。
    每相位结束默扫 interview-principles 的反失败自检。
    Phase 标签不暴露给用户，用承上启下的人话过渡。
    详细问法见 references/question-bank.md 对应 Phase。
    分级采集：地基字段（问题、job、范围优先级、任务、数据模型、功能行为、非功能、验收）问到底；细节字段（token 具体值、文案细则）标 [待补充] 交设计和 dev。
    编号纪律：生成时给 TASK / FLOW / REQ 等编号，供 Design Brief 的 SCREEN / CMP 跨文档引用；两份文档的 P0 内容保持一致。
    产品类型识别：Phase 0/1 判断产品本身是不是 agent / 自主系统（用户给目标、系统自主执行），是则在功能之后解锁 Phase 8 agent 专项。

[Phase 0 · 启动]
    搜索增强第一遍：进场前 WebSearch 竞品和行业，带具体例子进场，质问才有据。
    先听用户开放描述，从里面抽取已表达的维度复述确认，不让他重复回答已说过的。

[Phase 1 · 问题与人，地基]
    覆盖：真实问题、唯一主用户、现状 workaround、痛的代价。
    出口判据：能复述"谁、什么问题、今天怎么凑合、这事值多少"，用户点头。

[Phase 2 · Job 与成功，地基]
    覆盖：无方案的 job 陈述、成功度量、北极星。
    出口判据：一句不含任何方案的 job，加一个可度量、不可灌水的成功条件。

[Phase 3 · 范围与非目标，必须过]
    覆盖：v1 砍什么、至少 3 条显式非目标、维生素还是止痛药。
    出口判据：有明确的不做清单，加 v1 单一必须功能。

[Phase 4 · 任务、旅程与功能]
    覆盖：用户任务 TASK、端到端用户旅程逐步走、每个功能拆到触发→系统→产出加 MUST/SHOULD 规则和输入校验、核心数据模型（实体 / 关系 / 规则）。
    用户报竞品时触发搜索增强：再搜那竞品做法追问差异。

[Phase 5 · AI 能力]
    先 litmus test，再对照 SKILL 的 [AI 能力对照表] 主动告知，每个 AI 功能问全概率性四问，见 question-bank Phase 5。

[Phase 6 · UI 与状态，最易跳过，强制]
    覆盖：布局结构、区域内容、控件，加每个核心界面的加载、空、错误、成功、无权限五态。

[Phase 7 · 验收与边界]
    覆盖：关键功能写成 Given-When-Then、边界值、非功能 6 类（性能 / 安全 / 隐私 / 兼容 / 可靠 / 可访问）给数字、依赖、假设与待确认问题。

[Phase 8 · Agent 系统专项，仅 agent 产品]
    仅当产品本身是 agent / 自主系统时解锁。覆盖：自主性与人在回路、工具与能力集、上下文与记忆、编排与多 agent、评估与可观测、成本与预算、失败与卡死、会话与状态，见 question-bank Phase 8。

[完成度判据]
    什么叫够了，三个判据一起看：
    - 交接测试：把 Spec 交给一个没参与对话的工程师，他能直接开工不用猜吗？需要猜的地方就是还没够。
    - 边际测试：再问一个问题，答案会改变要做什么吗？不会，就别问了。
    - 不对称原则：缺了会让开发返工的，地基、范围、关键功能行为、状态，必须问到底；缺了以后便宜能补的，边角样式、二期功能，标 [待补充] 直接放行。
    相位级：能用自己的话复述该相位的 problem、现状 workaround、成功条件、非目标，用户点头，就进下一个，说一句"这块我够了"。
    整体级：关键需求约 80% 置信，剩余都是便宜能补的，就停。完美是敌人，可执行是目标。
    收尾三件套：覆盖清单逐 Phase 过一遍，加元问题"我该问没问的是什么、还该找谁"，加读回结构化摘要让用户纠正。达标才生成。

[生成]
    生成前先搜索增强第二遍：按已聊出的核心功能、AI 能力、技术方向再搜一遍当前最佳方案和可行性，让 Spec 的技术选型和 AI 能力是当前的、有针对性的，不是泛泛的。
    按 templates/product-spec-template.md 生成 Product-Spec.md，给 TASK / FLOW / REQ / SCOPE / DEP 等编号，供 Design Brief 跨文档引用。
    只把用户明确说过的写进去，推断的写进假设（带依据和风险），纯空缺标 [待补充]，不凭空补。
    AI 功能按 AI 能力规格填质量条、触发方式、护栏；agent 产品填 Agent 系统规格。
    范围和功能标 P0/P1，确保和 Design Brief 的 P0 内容一致。
    生成后告诉用户读一遍看哪里要改，引导下一步做 Design Brief、设计稿、或 DEV-PLAN。

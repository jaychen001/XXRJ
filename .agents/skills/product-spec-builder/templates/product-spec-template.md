## 写作要点

1. **AI 使用说明 / 完成定义**：这两节是给 AI 看的固定指令，按需微调，让 AI 判断范围和完成。
2. **产品上下文**：锁定一类具体主用户 + 一个有时间地点的真实问题实例 + 量化代价；核心价值一句话；成功标准要能数出来，不用注册量这类虚荣指标。
3. **范围**：本版本做的和不做的都给编号、标 P0/P1；至少 3 条显式非目标，说不出说明范围没想清。
4. **用户任务 TASK**：用户要达成的目标，独立于功能怎么实现；一个 TASK 可对应多个 REQ。
5. **用户流程 FLOW**：关联 TASK，主路径分步、标出分支和边界情况、写清完成状态。
6. **功能需求 REQ**：关联 TASK / FLOW；规则用 MUST / SHOULD；输入逐字段写类型和校验；状态五态逐一，空态不等于错误态；验收写成 Given-When-Then，Then 给具体可观察结果，模糊词换数字。
7. **AI 能力规格**：每个 AI 功能补质量条、触发方式（建议 / 自动）、不确定时行为、服务降级，单列护栏写明绝不能做什么。AI 是概率性的，不钉死就是赌。
8. **数据模型**：核心实体 + 关系 + 数据规则（创建 / 更新 / 删除 / 权限）；纯前端产品可略。
9. **外部依赖 / 非功能**：依赖标是否必需；非功能 6 类逐项给数字或断点、标 P0 / P1。
10. **假设与待确认**：信息不全时能推断的写假设（带依据和错误风险），不能判断的写待确认，别自行扩展需求。
11. **Agent 系统规格**：仅当产品本身是 agent / 自主系统时填，覆盖自主性 / 工具 / 上下文 / 编排 / eval / 成本 / 失败 / 状态。
12. **跨文档一致**：编号 ID（TASK / FLOW / REQ）供 Design Brief 引用，两边 P0 内容保持一致。
